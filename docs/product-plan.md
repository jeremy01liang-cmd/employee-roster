# 人事管理花名册产品方案

## 1. 产品定位

人事管理花名册用于统一维护员工基础档案，支持 HR、行政和管理者快速录入、查询、更新、删除人员信息，作为入转调离、组织架构、权限开通等流程的基础数据源。

## 2. 目标用户

- HR 专员：维护员工信息、办理入离职
- 行政人员：查看员工通讯录、工位/办公信息
- 部门负责人：按部门查看人员名单
- 系统管理员：配置权限、审计操作记录

## 3. 核心目标

- 让人员信息录入与维护标准化
- 让花名册查询足够快、字段足够全
- 为后续薪资、考勤、审批、组织架构模块提供基础数据

## 4. 功能范围

### 4.1 核心功能

1. 人员列表
2. 新增人员
3. 编辑人员
4. 删除人员
5. 搜索与筛选
6. 状态管理（在职、离职）

### 4.2 建议扩展

1. 批量导入/导出 Excel
2. 组织架构树
3. 入离职流程联动
4. 操作日志
5. 角色权限控制
6. 附件上传（合同、身份证明等）

## 5. 业务流程

### 5.1 新增人员

1. HR 点击“新增人员”
2. 填写工号、姓名、性别、手机号、邮箱、部门、岗位、入职日期等信息
3. 系统校验必填项、工号唯一性、手机号和邮箱格式
4. 保存成功后进入花名册列表

### 5.2 编辑人员

1. 在列表中选择一名员工
2. 点击“编辑”
3. 修改人员信息
4. 系统保留更新时间

### 5.3 删除人员

1. 在列表中点击“删除”
2. 二次确认
3. 删除记录

说明：生产环境更建议改为“逻辑删除”或“转离职状态”，避免历史数据丢失。

## 6. 页面设计

## 6.1 花名册列表页

页面区域：

- 顶部标题区：系统名称、模块说明
- 搜索区：姓名/工号/部门关键字搜索
- 操作区：新增人员按钮
- 表格区：展示员工列表
- 行操作：编辑、删除

表格字段建议：

- 工号
- 姓名
- 性别
- 手机号
- 邮箱
- 部门
- 岗位
- 入职日期
- 状态
- 紧急联系人
- 紧急联系人电话
- 备注
- 更新时间

## 6.2 新增/编辑弹窗

表单字段：

- 工号（必填，唯一）
- 姓名（必填）
- 性别（选填）
- 手机号（必填）
- 邮箱（选填）
- 部门（必填）
- 岗位（必填）
- 入职日期（必填）
- 状态（默认在职）
- 出生日期（选填）
- 学历（选填）
- 身份证号（敏感字段，生产环境建议脱敏与加密）
- 家庭住址（选填）
- 紧急联系人（选填）
- 紧急联系人电话（选填）
- 备注（选填）

## 7. 字段规则

### 7.1 数据校验

- 工号唯一，不可重复
- 姓名长度 2 到 20
- 手机号满足中国大陆手机号规则
- 邮箱满足标准邮箱格式
- 入职日期不能为空
- 部门、岗位不能为空

### 7.2 状态定义

- `active`：在职
- `inactive`：离职

## 8. 权限设计

### 8.1 角色建议

- `admin`：全量增删改查
- `hr`：全量增删改查
- `manager`：查看本部门、编辑有限字段
- `viewer`：只读

### 8.2 权限点

- `employee:view`
- `employee:create`
- `employee:update`
- `employee:delete`

## 9. 前端设计

技术方案：

- React + TypeScript + Vite
- 单页应用
- 使用原生 `fetch` 调用 REST API
- 通过弹窗表单完成新增和编辑

前端模块拆分：

- `App.tsx`：列表页容器
- `styles.css`：页面样式
- `types.ts`：员工类型定义

前端状态设计：

- `employees`：员工列表
- `query`：搜索关键字
- `formData`：新增/编辑表单数据
- `editingEmployee`：当前正在编辑的员工
- `submitting`：提交中状态

## 10. 后端设计

技术方案：

- Node.js + Express
- RESTful API
- `better-sqlite3` 作为数据库访问层
- `cors` 支持前后端分离开发

接口设计：

### 10.1 查询员工列表

- `GET /api/employees?q=关键字`
- 支持按姓名、工号、部门模糊搜索

### 10.2 新增员工

- `POST /api/employees`

请求体示例：

```json
{
  "employeeCode": "A1024",
  "name": "张三",
  "gender": "男",
  "phone": "13800138000",
  "email": "zhangsan@example.com",
  "department": "产品部",
  "position": "产品经理",
  "status": "active",
  "hireDate": "2026-03-18"
}
```

### 10.3 编辑员工

- `PUT /api/employees/:id`

### 10.4 删除员工

- `DELETE /api/employees/:id`

返回格式建议：

```json
{
  "success": true,
  "message": "ok",
  "data": {}
}
```

## 11. 数据库设计

数据库：SQLite（开发阶段）

生产可迁移到 MySQL / PostgreSQL。

### 11.1 表结构

表名：`employees`

| 字段名 | 类型 | 说明 |
|---|---|---|
| id | INTEGER | 主键，自增 |
| employee_code | TEXT | 工号，唯一 |
| name | TEXT | 姓名 |
| gender | TEXT | 性别 |
| phone | TEXT | 手机号 |
| email | TEXT | 邮箱 |
| department | TEXT | 部门 |
| position | TEXT | 岗位 |
| status | TEXT | 状态 |
| hire_date | TEXT | 入职日期 |
| birth_date | TEXT | 出生日期 |
| education | TEXT | 学历 |
| id_number | TEXT | 身份证号 |
| address | TEXT | 地址 |
| emergency_contact | TEXT | 紧急联系人 |
| emergency_phone | TEXT | 紧急联系人电话 |
| notes | TEXT | 备注 |
| created_at | TEXT | 创建时间 |
| updated_at | TEXT | 更新时间 |

### 11.2 索引设计

- 唯一索引：`employee_code`
- 普通索引建议：`name`、`department`、`status`

## 12. 非功能要求

- 支持 1 万级员工数据基础查询
- 表单校验响应时间小于 200ms
- 列表接口响应时间小于 500ms
- 敏感字段需要脱敏展示
- 删除操作必须具备确认机制
